# 反模式清单 (Anti-Patterns)

以下为本 Skill 产出物中禁止出现的反模式。AI 在生成任何 Step 的产出物后，
必须对照此清单自检。

## 一、User Story 反模式

| 反模式 | 示例 | 正确做法 |
|--------|------|---------|
| AC 使用模糊动词 | "正确处理错误" | "显示红色提示'灵气不足'" |
| AC 缺少 Given | "当点击按钮，显示结果" | "Given 弟子在采集，When Tick 触发…" |
| Story 过大 | "实现整个弟子系统" | 拆分为独立的子 Story |
| 缺少边界 AC | 只有正常路径 | 增加异常/边界 AC |
| 依赖不明确 | "需要其他系统支持" | "依赖 Story #3（弟子状态机）" |

## 二、数值设计反模式

| 反模式 | 示例 | 正确做法 |
|--------|------|---------|
| 无公式凭感觉 | "产出大概 10-20 灵气" | "产出 = base × (1 + 弟子等级 × 0.1)" |
| 无 Sink 兜底 | "弟子无限产出灵草" | 设计灵草消耗（炼丹）和保质期上限 |
| 编造 Baseline | "假设基础灵气为 100" | 查询现有 `shared/formulas/` 获取真实值 |

## 三、架构反模式

| 反模式 | 示例 | 正确做法 |
|--------|------|---------|
| 跨层直接调用 | UI 层直接修改 GameState | 通过 Engine 层的事件/Action |
| 逻辑写在 client/ | Electron 进程中计算收益 | 写在 game/ 或 shared/ |
| 公式散落各处 | 三个文件里各写一遍倍率 | 公式统一在 shared/formulas/ |

## 四、流程控制反模式

| 反模式 | 示例 | 正确做法 |
|--------|------|---------|
| 跳过 Phase 门禁 | "Phase I 差不多了，先做 Phase II" | HARD-GATE 要求 USER 明确确认后才能进入下一 Phase |
| 验证结论用模糊词 | "应该没问题"、"大概通过了" | 必须基于实际运行结果陈述事实 |
| 跳过 Self-Review | "内容不多，不用自检" | 每个 Phase 完成后必须执行 Self-Review |
| 未问就猜 | "假设用户的意思是…" | 遇到歧义立即停下来问 USER |
| 一次性输出多步 | 把 Step 1-3 合成一条消息 | 每次只输出 1 个 Step，末尾抛选择题 |
| 跳过 PoC 验证 | "这个库很成熟，直接用" | 未经验证的技术方案必须走 Step 1.5 PoC |

## 五、AI 集成反模式

| 反模式 | 示例 | 正确做法 |
|--------|------|---------|
| AI 决定数值 | 让 AI 模型决定伤害、收益等关键数值 | AI 只负责表现层（台词、描述），数值由确定性引擎计算 |
| 无降级方案 | AI 服务不可用时游戏卡死 | 设计 fallback 路径，AI 不可用不阻塞核心玩法循环 |
| 上下文无边界 | 把全量游戏状态丢给 AI | 定义最小必要上下文，控制输入规模和成本 |
| 跳过 PoC 直接集成 | "这个模型/库很成熟，直接用" | 关键技术假设必须通过 Step 1.5 PoC 验证 |
| AI 输出未校验 | 直接展示模型原始输出 | 对 AI 输出做格式校验和内容过滤，防止破坏游戏沉浸感 |
