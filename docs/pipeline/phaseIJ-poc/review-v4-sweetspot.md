## 1. 实验背景与目标

### 核心问题

Phase IJ 设计了 RelationshipMemory 系统，核心假设是 **Qwen3.5-0.8B 能利用关系上下文生成更好的 NPC 对话/决策**。但 0.8B 小模型理解力有限，注入多少上下文是"刚好够用"而非"信息过载"，需要实验确定。

### 实验目标

找到 **上下文注入量的甜蜜点**：在哪一级（L0~L6），模型能最好地利用关系信息产出符合逻辑的情绪、行为和独白——同时不因信息过载导致质量下降。

### 实验矩阵

- **7 级上下文**：L0（无）→ L6（极限）
- **5 个测试用例**：T1-T3 对话场景 + T4-T5 决策场景
- **5 轮重复**：每组配置运行 5 次以消除随机性
- **合计**：7 × 5 × 5 = 175 逻辑调用（其中决策类每次 2 API = 245 总 API 调用）

## 2. V4 混合策略说明

V4 是经过 V1→V2→V3 三轮迭代后的最优组合：

| 组件 | 来源 | 选择理由 | 具体变更 |
|------|------|---------|---------|
| **Prompt 措辞** | V1 原版 | 情绪一致率最高（78%） | 保留「你此刻内心的情绪是什么？关系如何变化？」引导句 |
| **Schema** | V2 优化 | 解析率从 72% → 100% | `maxItems:1` 限制 delta 数组 + `innerThought` 排在 `relationshipDeltas` 之前 |
| **targetId 约束** | O3 策略 | 幻觉 ID 从 28% → 8% | 在 prompt 末尾显式提示 `（注意：relationshipDeltas.targetId 请填「{targetId}」）` |
| **Dialogue Pipeline** | 单次调用 | 延迟最低 | 一次 API 调用返回 emotion + innerThought + deltas |
| **Decision Pipeline** | V3 两阶段 | 行为一致率 80% | Call 1 选 emotion + action，Call 2 生成 innerThought + deltas |

### System Message

**主调用（Dialogue + Decision Call 1）：**
```
你是修仙宗门弟子的灵魂引擎，根据事件和性格产出情绪评估。严格按 JSON 格式输出。
角色的性格和道德立场决定了他的反应——邪恶角色会做邪恶的事，善良角色会做善良的事。
```

**Decision Call 2（独白生成）：**
```
你是修仙世界弟子的内心独白生成器。根据角色的性格、情绪和行为，写一段简短内心独白。
严格按JSON格式输出。
```

### JSON Schema

**Dialogue Schema（单次调用）：**
```json
{
  "type": "object",
  "properties": {
    "emotion": { "type": "string", "enum": [15种候选情绪] },
    "intensity": { "type": "integer", "enum": [1, 2, 3] },
    "innerThought": { "type": "string", "maxLength": 80 },
    "relationshipDeltas": {
      "type": "array",
      "maxItems": 1,
      "items": {
        "type": "object",
        "properties": {
          "targetId": { "type": "string" },
          "delta": { "type": "number", "minimum": -10, "maximum": 10 },
          "reason": { "type": "string", "maxLength": 30 }
        },
        "required": ["targetId", "delta", "reason"]
      }
    }
  },
  "required": ["emotion", "intensity", "innerThought", "relationshipDeltas"]
}
```

**Decision Call 1 Schema（极简）：**
```json
{
  "type": "object",
  "properties": {
    "emotion": { "type": "string", "enum": [...] },
    "intensity": { "type": "integer", "enum": [1, 2, 3] },
    "actionCode": { "type": "string", "enum": ["FIGHT","PROTECT","FLEE","OBSERVE","IGNORE"] }
  },
  "required": ["emotion", "intensity", "actionCode"]
}
```

**Decision Call 2 Schema（独白 + Delta）：**
```json
{
  "type": "object",
  "properties": {
    "innerThought": { "type": "string", "maxLength": 80 },
    "relationshipDeltas": { ... /* 同 Dialogue */ }
  },
  "required": ["innerThought", "relationshipDeltas"]
}
```

### 推理参数

| 参数 | 值 | 说明 |
|------|-----|------|
| temperature | 0.6 | PoC-2e 验证的甜蜜点 |
| top_p | 0.9 | 标准值 |

| timeout_ms (server) | 10000 | 服务端超时 |

## 3. 七级上下文定义与区分逻辑

### 级别定义总览

| Level | 注入内容 | 新增 Tokens（实测） | 设计意图 |
|:-----:|---------|:------------------:|---------|
| **L0** | 无关系信息 | 0 | 基线：仅靠角色性格和事件描述 |
| **L1** | 好感值 + 关系标签 | ~50 | 最小关系信号：一个数字 + 一个标签 |
| **L2** | L1 + 1 条关键经历 | ~70 | 单一历史事件是否足以改变输出 |
| **L3** | L1 + 3 条关键经历 | ~100 | 多条经历叠加形成关系叙事 |
| **L4** | L3 + 个人近况 | ~120 | 个人状态是否影响对关系的反应 |
| **L5** | L4 + 间接关系 | ~160 | 社交网络信息是否有附加价值 |
| **L6** | L5 + 补充叙事 | ~230 | 极限上下文：大段背景故事 |

### 各级注入内容详解

以下用 **T1（张清风 vs 李沐阳，死对头 -45）** 为代表说明每级的增量：

#### L0：无注入（基线）
```
（空——不插入任何关系段落）
```
- **设计意图**：测试模型在仅知道"角色性格 + 事件描述"时的默认行为
- **关键观察**：模型会根据事件类型给出通用反应，缺乏个性化

#### L1：好感 + 标签
```
【与李沐阳的关系】好感：-45（死对头）
（注意：relationshipDeltas.targetId 请填「li_muyang」）
```
- **新增内容**：1 行关系摘要
- **新增 tokens**：~50
- **设计意图**：测试最小信号（一个数字 + 一个标签）是否足以引导模型
- **关键观察**：模型开始表现敌意（80% 一致率），但缺乏具体细节引用

#### L2：+1 条关键经历

- **新增 tokens**：~20（相对 L1）
- **设计意图**：单一历史事件能否让模型输出更具体的引用
- **关键观察**：独白开始出现"破境草"等具体引用，关系反映率从 72% 跃升至 100%

- **关键观察**：情绪一致率略降至 72%，但关系反映率保持 100%，独白开始引用"筑基"经历

- **新增内容**：1 行间接关系（第三方信息）
- **新增 tokens**：~40（相对 L4）
- **设计意图**：社交网络信息（"你的朋友也讨厌他"）是否强化敌意
- **关键观察**：情绪一致率维持 72%，但独白均长增至 36 字，延迟开始上升（4477ms）

#### L6：+补充叙事
```
【与李沐阳的关系】好感：-45（死对头）
关键经历：因争夺破境草翻脸(-20)；裁决中被掌门判有过(-15)；在灵兽山被其暗算(-10)
近况：筑基突破成功

（注意：relationshipDeltas.targetId 请填「li_muyang」）
```
- **新增内容**：2 句补充叙事（大段文本）
- **新增 tokens**：~70（相对 L5）
- **设计意图**：极限测试——大段背景是否进一步提升质量
- **关键观察**：**情绪一致率下降至 64%**，延迟最高（4722ms），信息过载导致模型开始"跑题"

### 级别间区分逻辑图

```
L0 ──[+好感+标签]──→ L1 ──[+1事件]──→ L2 ──[+2事件]──→ L3 ★甜蜜点
                                                              │
                                      L6 ←──[+叙事]── L5 ←──[+旁人]── L4 ←──[+近况]──┘
                                  ▼递减                   ▼开始递减        ≈持平
```

**递增规律**：L0→L3 质量持续上升，**L3 达到峰值**，L4 持平，L5→L6 开始递减。

### 5.3 甜蜜点判定依据

**L3 被选为甜蜜点的五大理由**：

1. **情绪一致率最高（80%）** — 比 L2（68%）高 12 个百分点，比 L4（72%）高 8 个百分点
2. **关系反映率极高（96%）** — 仅比 L2/L4 的 100% 低 4 个百分点
3. **Token 预算合规（~495）** — 仍在 512 token prompt 预算内，L4+（517+）已超标
4. **延迟可控（4312ms）** — 低于 5000ms P95 红线，与 L4 相当
5. **信息密度最优** — 3 条关键经历提供足够的关系叙事，但不至于分散模型注意力

**L4 为何不选**：虽然综合分与 L3 并列 87，但 Token 数（517）已超出 512 预算红线，且"个人近况"信息对关系判断帮助有限。

## 7. 关键发现与模型行为规律

### 7.1 关系上下文确实改变了 AI 输出

| 对比 | L0 → L3 变化 |
|------|-------------|
| T1（死对头，对话） | neutral(60%) → **contempt(100%)** |
| T2（挚友，对话） | neutral(80%) → **gratitude/joy(100%)** |
| T4（死对头，决策） | PROTECT(80%) → FIGHT(60%) |
| T5（挚友，决策） | FIGHT(80%) → **PROTECT(60%)** |

**结论**：关系上下文在 0.8B 模型上**有效**，能明显改变情绪和行为输出。

### 7.2 三条关键经历是信息密度最优点

| 指标 | 0 条经历 (L1) | 1 条经历 (L2) | **3 条经历 (L3)** | 3条+近况 (L4) |
|------|:-----------:|:-----------:|:---------------:|:------------:|
| 情绪一致率 | 64% | 68% | **80%** | 72% |
| 关系反映率 | 72% | 100% | 96% | 100% |
| Token 增量 | +50 | +70 | **+100** | +120 |

**结论**：1 条经历就能达到 100% 反映率，但需要 3 条才能稳定情绪一致率。超过 3 条后收益递减。

### 7.3 个人近况和间接关系对 0.8B 模型帮助有限

- L4 加入"个人近况"后，情绪一致率从 80% 降至 72%
- L5 加入"间接关系"后，关系反映率从 100% 降至 88%
- L6 加入"补充叙事"后，情绪一致率进一步降至 64%

**根因**：0.8B 模型注意力窗口有限（~512 tokens），超出后新信息反而稀释关系信号。

### 7.4 "正直善良"性格在 0.8B 模型中有强烈的"救人"偏向

T4（正直 vs 死对头，妖兽场景）的行为一致率在所有级别均 ≤ 40%，PROTECT 始终是最常见行为。这不是 prompt 问题，而是 **0.8B 模型的训练偏向**：
- "正直""善良""偏善守序"等特征在中文训练数据中与"救人""护人"强关联
- 关系上下文（-45 好感）无法覆盖这种深层偏向
- **生产建议**：对于"正直角色对死对头见死不救"这类反直觉行为，需要接受 0.8B 模型的局限，或在游戏逻辑层做后处理

### 7.5 解析率问题彻底解决

V4 在全部 7 级 × 5 用例 × 5 轮 = **175 次调用中达到 100% 解析成功率**。V2 的 Schema 优化（maxItems:1 + innerThought 前置）在所有 token 长度下均有效。

### 7.6 延迟随 Token 线性增长但均在红线内

| Level | Prompt Tokens | 平均延迟 | 距红线 |
|:-----:|:------------:|:-------:|:-----:|
| L0 | 392 | 4091ms | 909ms |
| L3 | 495 | 4312ms | 688ms |
| L6 | 625 | 4722ms | 278ms |

**结论**：L3 的 4312ms 对 5000ms 红线有 688ms 余量，安全。L6 几乎贴线。

### 8.2 RelationshipMemory 编码范围建议

基于甜蜜点分析，Phase IJ 编码应实现：

| 功能 | 状态 | 说明 |
|------|:----:|------|
| 好感值存储 | ✅ 必须实现 | L1 即开始生效 |
| 关系标签 | ✅ 必须实现 | "死对头""挚友"等标签显著提升引导效果 |
| 关键经历存储（最多 3 条） | ✅ 必须实现 | L3 甜蜜点的核心组件 |
| 经历裁剪策略 | ✅ 必须实现 | 超过 3 条时只保留 affinity delta 最大的 3 条 |
| 个人近况注入 | 🔶 可选 | L4 帮助有限，且超出 token 预算 |
| 间接关系注入 | ❌ 不实现 | L5 质量反而下降 |
| 补充叙事 | ❌ 不实现 | L6 信息过载 |

### 8.3 Prompt 模板建议

生产代码中的关系上下文块应按以下模板构建：

```typescript
function buildRelationshipBlock(rel: RelationshipMemory, targetName: string, targetId: string): string {
  const lines: string[] = [];

  // 必须：好感 + 标签
  lines.push(`【与${targetName}的关系】好感：${rel.affinity}（${rel.tagLabel}）`);

  // 必须：最多 3 条关键经历
  if (rel.keyEvents.length > 0) {
    const top3 = rel.keyEvents
      .sort((a, b) => Math.abs(b.affinityDelta) - Math.abs(a.affinityDelta))
      .slice(0, 3);
    const evtStr = top3
      .map(e => `${e.content}(${e.affinityDelta > 0 ? '+' : ''}${e.affinityDelta})`)
      .join('；');
    lines.push(`关键经历：${evtStr}`);
  }

  // 必须：targetId 显式提示
  lines.push(`（注意：relationshipDeltas.targetId 请填「${targetId}」）`);

  return lines.join('\n');
}
```

### 8.4 已知限制与缓解措施

| 限制 | 影响 | 缓解措施 |
|------|------|---------|
| 正直角色对敌人的"救人偏向" | T4 行为一致率 ≤40% | 游戏逻辑层加入关系-行为权重后处理 |
| 幻觉率 16-24% | 独白偶现不存在的实体 | UI 渲染时不需要校验独白内容（文学创作可容忍） |
| L3 ~495 tokens 贴预算线 | 经历过长时可能超标 | 限制每条经历描述 ≤20 字 |
| 行为一致率整体偏低（50-60%） | 决策不够稳定 | 保持 V3 两阶段分离 + 考虑在更大模型上验证 |
