## §2 第一性原理解构 + 价值锚定

### 2a. 5-Why 根因链

1. **为什么需要面板？** → 因为 look/inspect/sect/judge 的输出插入日志流后会被冲走
2. **为什么被冲走是问题？** → 因为这些是高信息密度内容（弟子五维/道德/关系/特性），玩家需要持续参考
3. **为什么需要持续参考？** → 因为玩家做决策（裁决/突破/观察弟子变化）时需要同时看到弟子状态
4. **为什么日志流不能满足？** → 因为日志流是时间线性的，新消息不断涌入，旧内容物理位移消失
5. **根因** → **结构化信息需要独立的展示通道，不应混入时序日志流**

### 2b. Invariant 声明

| # | 不变量 | 违反后果 |
|---|--------|---------|
| I1 | 面板显示不阻塞日志流更新 | 打开面板时世界停止更新 → 挂机核心体验崩塌 |
| I2 | 面板覆盖层不拦截命令输入 | 输入框失焦 → 操作断裂 |
| I3 | 同时只能有一个面板实例 | 多面板叠加 → DOM 泄漏 + 界面混乱 |
| I4 | 面板内容为 formatter 纯函数的直接产出 | 面板不能引入新的数据格式化逻辑 → 避免职责分裂 |
| I5 | ESC/点击背景可关闭面板 | 退出路径不统一 → UX 碎片化 |

### 2c. 最小组件检验

> 如果只保留一个核心功能：**inspect 面板不随日志滚走**。
> 原因：inspect 是弟子灵魂档案的唯一入口，信息密度最高（5 区块 × 多行），被冲走后完全无法阅读。

### 2d. 核心体验锚定

| 维度 | 评估 |
|------|------|
| **核心体验** | "掌门操控如指臂使" — 结构化信息触手可及，不再在日志流中追逐 |
| **ROI 评估** | 开发成本 **S**（纯 DOM overlay + 已有 formatter）vs 体验增量 **4/5**（解决 X-05 痛点） |
| **循环挂载** | 观察循环（look/inspect/sect）+ 干预循环（judge裁决）= 玩家信息获取的全部入口 |
| **约束** | 纯 Presentation 层，零 Engine/AI/Data 改动，HC-1~HC-5 继承 X-β |

| # | 约束 | 理由 |
