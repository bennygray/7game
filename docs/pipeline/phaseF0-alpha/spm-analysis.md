## Step 1 记录：第一性原理解构

### 5-Why 根因链

1. 为什么要做碰面世界？→ 8弟子是互不可见的平行线，灵魂数据无法被验证
2. 为什么灵魂数据无法验证？→ 数据要在社交场景中才产生可观测差异，当前引擎无社交场景
3. 为什么无社交场景？→ 弟子没有空间概念，不可能"碰到谁"
4. 为什么需要空间？→ 不需要真正拓扑，只需逻辑分组让同行为弟子归入同"地点"
5. **根因**：最低成本的逻辑分组 + 碰面检定 = 灵魂系统的验证基础设施

## L2 CoVe 对 WARN 项验证

### WARN-1（R3-D2 极端模拟：碰面可能偏多）

**验证问题**：
1. 8 弟子全部修炼的概率有多大？→ 五维权重中 MEDITATE 权重约 0.35，8 弟子同时修炼概率 ≈ 0.35^8 ≈ 0.02%。极端情况极罕见。
2. 实际典型场景中同地弟子数？→ 平均 2~3 对同地 → 每 5 秒 0.4~0.6 次碰面 → 每分钟 5~7 次碰面 → 冷却过滤后 2~4 次有意义碰面/分钟。
3. 2~4 次/分钟是否需要 BLOCK？→ 每分钟 2~4 条碰面日志可接受，混在其他行为日志中占比低。

**CoVe 结论**：WARN 维持。极端场景罕见，典型场景下密度可接受。SGE 测试后可微调常量。

### WARN-2（R4-D5 交付验证：碰面密度难自动化）

**验证问题**：
1. 是否真的无法自动化？→ 可编写统计脚本模拟 N 轮 tick，断言碰面事件数/分钟在预期范围 [2, 15]
2. 这本质上是一个数值调优问题，是否应阻塞？→ 否，常量可调，回归测试覆盖存档迁移即可

**CoVe 结论**：WARN 维持。添加统计脚本到验证计划。
