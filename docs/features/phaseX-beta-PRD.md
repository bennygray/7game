## §1 第一性原理解构

### 1a. 5-Why 根因链

```
Q1: 为什么需要命令增强？
A:  X-α 已解决信息分层，但命令交互仍是 Phase A 原型级——
    无补全、无视觉反馈、无别名扩展。

Q2: 为什么命令交互重要？
A:  MUD 的唯一交互通道就是命令输入框。交互效率直接决定
    玩家能否顺利使用 Phase H~X-α 新增的 10 个命令。

Q3: 为什么现在做？
A:  X-α 已完成拆分（command-handler.ts 独立），
    新功能可以在已有模块上叠加，零架构风险。

Q4: 为什么不等到 X-γ 一起做？
A:  X-γ 是面板系统（浮层 UI），复杂度 L。命令增强和视觉反馈
    是独立的 S~M 改进，先交付能立刻提升操作效率。

Q5: 什么是最根本的问题？
A:  **操作效率** — 玩家已经能"看到"世界（X-α），
    但还不能"高效操作"世界。命令补全和视觉反馈是效率的最后一公里。
```

**根因**：展现层的信息层级已解决（X-α），但操作效率和视觉反馈仍停留在原型级，导致玩家与"活世界"的交互体验不匹配。

### 1b. Invariant 声明

| # | 不变量 | 违反后果 |
|---|--------|---------| 
| I1 | 展现层改造不得修改 Engine/AI/Data 层的任何代码 | 引擎回归风险 |
| I2 | GameState v5 不变，零存档迁移 | 存档兼容性 |
| I3 | 回归测试 64/64 必须持续通过 | 系统稳定性 |
| I4 | 不引入重 DOM 框架（React/Vue/Svelte）或外部补全库 | Electron 迁移预期 |
| I5 | MUD 文字基调保持 | 产品定位 |
| I6 | 所有现有命令功能不丢失，现有别名（l/i/bt）不变 | 向后兼容 |
| I7 | main.ts 保持 ≤ 150 行 | X-α 已交付约束 |

### 1c. 最小组件检验

> 如果只保留一个核心功能，是哪个？

**Tab 自动补全**。8 个弟子名 + 10 个命令，纯键盘零记忆负担。

### 1d. 核心体验锚定

| 维度 | 内容 |
|------|------|
| **核心体验** | 掌门按 Tab 即可补全弟子名和命令，突破时屏幕闪烁，行为日志一目了然 |
| **开发成本** | S~M（小~中）— 纯 Presentation 层叠加，复用 X-α 架构 |

## §2 规则与数值边界

### §2.1 业务实体

本 Phase 不引入新的 GameState 实体。改造的是**命令交互层**：

| 概念 | 定义 | 持久化 |
|------|------|:------:|
| **补全候选池（CompletionPool）** | Tab 候选列表 = 命令名 + 弟子名 | ❌ |
| **命令别名表（AliasMap）** | 短别名 → 完整命令的映射 | ❌ |
| **行为图标表（BehaviorIconMap）** | 行为 ID → Unicode emoji 的映射 | ❌ |
| **突破闪烁效果（BreakthroughFlash）** | CSS @keyframes 动画 | ❌ |

### §2.2 Tab 自动补全规则

> [!IMPORTANT]
> **这是 Phase X-β 的核心规则——Tab 补全行为定义。**

#### T-补全规则

| 规则ID | 条件 | 行为 | 说明 |
|--------|------|------|------|
| T-01 | 输入框为空 + 按 Tab | 无动作 | 空输入不触发补全 |
| T-02 | 输入 `lo` + 按 Tab | 补全为 `look ` | 唯一命令匹配，追加空格 |
| T-03 | 输入 `look 张` + 按 Tab | 补全为 `look 张清风` | 唯一弟子匹配 |
| T-04 | 输入 `look ` + 按 Tab | 循环弟子名列表 | 多匹配连续 Tab 循环 |
| T-05 | 输入 `s` + 按 Tab | 补全 `sect` → 再 Tab → `status` | 多命令匹配，连续 Tab 循环 |
| T-06 | 输入 `xyz` + 按 Tab | 无动作 | 无匹配 |
| T-07 | 补全后按其他键 | 退出循环模式 | 恢复正常输入 |
| T-08 | Shift+Tab | 反向循环 | 与 Tab 方向相反 |

#### 补全候选池构成

| 来源 | 内容 | 数量 | 示例 |
|------|------|:----:|------|
| 命令名 | 所有可用命令的 verb | 10 | `look`, `inspect`, `sect`, `judge`, `status`, `bt`, `clear`, `reset`, `ai`, `help` |
| 弟子名 | `state.disciples[].name` | 8 | `张清风`, `李墨言`, ... |

#### 补全逻辑

```
输入 = "look 张"
  ↓
分割: verb="look", argPrefix="张"
  ↓
verb 已完成 → 进入参数补全
  ↓
过滤弟子名：startsWith("张") → ["张清风"]
  ↓
唯一匹配 → 直接填入 "look 张清风"
```

```
输入 = "s"
  ↓
verb 前缀匹配：["sect", "status"]
  ↓
多匹配 → Tab 循环：sect → status → sect → ...
```

#### CSS 动画定义

```css
/* 突破成功 — 金色闪烁 */

  0%   { opacity: 0; }

  30%  { opacity: 0.1; }
  50%  { opacity: 0.35; }
  70%  { opacity: 0.05; }
  100% { opacity: 0; }
}

/* 突破失败 — 红色闪烁 */

  0%   { opacity: 0; }

  100% { opacity: 0; }
}
```

## §6 Pre-Mortem 分析

### 假设

> 假设命令增强功能已上线 3 个月，结果是**彻底失败**——玩家不用、体验变差。

### 失败原因分析

| # | 最可能的失败原因 | 预警信号 | 缓解措施 | 当前风险等级 |
|---|----------------|---------|---------|:----------:|
| 1 | Tab 补全循环顺序不直觉（弟子名排序不稳定） | 玩家反馈"每次 Tab 出来的不一样" | 候选池排序：命令按字母序、弟子按 state.disciples 数组顺序（固定） | 🟢 低 |
| 2 | emoji 图标在不同终端/字体下宽度不一致，导致日志对齐混乱 | 终端/Electron 显示错位 | 选用 Unicode 基本面 emoji（有最广兼容性），且图标后跟空格固定宽度 | 🟡 中 |
| 3 | 突破闪烁效果在高对比度/光敏场景下造成不适 | 无障碍投诉 | 闪烁最大 opacity: 0.4（非全白），且可通过 CSS 变量 `--mud-flash-opacity` 禁用 | 🟢 低 |

### 行动项

> 🟡 的 emoji 宽度问题在 SGE 实施时验证，如果不一致，改为纯文字括号前缀 `[修]`/`[炼]`/`[探]`。
