## §1 第一性原理解构

### 1.0 需求债务检查

| 关联 FB | 描述 | 处理 |
|---------|------|------|
| **FB-005** | 关系面板 UI（非 MUD 日志的主动查看入口） | ✅ **本次清偿**（`look <弟子>` + `rel` 命令） |
| **FB-012** | 对话实体缺失（NPC 行为需可被听到的对外发声） | 🔶 **部分清偿**（环境描写 + 日志分级让世界有声音） |
| **FB-011** | 玩家干预权缺失 | ⬜ **不在本次范围**（Phase H 后续片段） |

### 1a. 5-Why 根因链

```
Q1: 为什么需要 MUD 世界呈现升级？
A1: 因为引擎已经有丰富数据（6 Zone、12 事件、碰面、情绪、关系），但玩家看到的只是一个无差别的日志流水账。

Q2: 为什么日志流水账不够？
A2: 因为流水账没有空间感（不知道弟子在哪）、没有节奏感（所有事件同等级显示）、没有交互性（不能主动查看某个弟子或地点）。

Q3: 为什么空间感/节奏感/交互性重要？
A3: 因为这三者构成了"活世界"的最低感知阈值——没有它们，灵魂系统（Phase E/F）和世界骨架（Phase F0）的数据
    就是不可感知的死数据。

Q4: 为什么不可感知的数据是问题？
A4: 因为产品愿景（MASTER-PRD §1.1）是"每个弟子都有灵魂的活世界"——如果灵魂和世界不可感知，核心差异化就不存在。

Q5: 为什么现在做而不是等 Phase G/H？
A5: 因为 Phase G（AI 觉醒）生产的 AI 独白和行为决策也需要"被看到"的载体。如果表现层仍然是流水账，
    Phase G 的产出同样不可感知。表现层升级是 Phase G 和 Phase H 的共同前置。
```

**根因价值**：表现层是引擎数据→玩家体验的唯一桥梁。桥断了，一切引擎工作的投入都无法变现。

### 1b. Invariant 声明

| # | 不变量 | 违反后果 |
|---|--------|---------|
| I1 | **表现层只读 GameState** — `main.ts` 不得修改任何引擎状态 | 违反 MASTER-ARCH X-1，表现层与引擎耦合 |
| I2 | **地点是行为的纯函数** — 观察视角切换不改变弟子实际位置 | 与 Phase F0-α 的 I1 冲突，地点系统被污染 |
| I3 | **零引擎改动** — 本 Phase 不新增/修改任何 Handler、Engine、Data 层代码 | 违反则需 SGA 审查 + 存档迁移 |
| I4 | **日志分级映射 EventSeverity 枚举** — 不得引入竞争的分级体系 | 表现层与引擎的分级标准分裂 |
| I5 | **MUD 纯文字约束** — 不引入 Canvas/SVG/图片等非文字渲染 | 违反 MASTER-PRD §4.1 IN/OUT 边界 |

### 1c. 最小组件检验

**如果只保留一个核心功能**：`look` 命令 — 让玩家能以场景为单位查看弟子和事件。

**理由**：`look` 是经典 MUD 的第一命令（参考北大侠客行/DikuMUD），它将"拉式查看"引入当前"推式流水账"，
让信息获取从被动变主动。这一个改变就能让世界从"墙纸"变成"可探索的空间"。

### 1d. 核心体验锚定

| 维度 | 值 |
|------|---|
| **核心体验** | "走进宗门，四处看看，弟子各有各的忙碌" — 空间感 + 生活感 |
| **ROI 评估** | 开发成本 **S**（纯表现层，零引擎改动） / 体验增量 **4/5** |
| **循环挂载** | 表现层是 §2 核心体验闭环的最终输出端（弟子行为→日志→玩家感知） |

**USER 决策（2026-03-30 确认）**：

1. ✅ **观察视角**：仅 `look` 命令，不做空间导航（n/s/e/w）— 当前阶段是基础搭建期，保持简单
2. ✅ **环境呼吸**：全局共用一套文案 — 弟子人数少，不需要 Zone 级分化
3. ✅ **日志默认模式**：全量显示 — 当前处于调试阶段，需要完整信息
