## §1 产品定位

### 1.1 核心体验

宗门中有男有女，叙事自然区分性别，为未来情感关系系统奠基。

### 1.2 5-Why 根因链

```
为什么需要性别系统？
→ 因为所有弟子隐性为男性，宗门世界感觉单调失真
  → 为什么单调失真重要？
  → 因为修仙宗门的核心幻想是"一群性格各异的人共处修行"，缺性别就少了一半人
    → 为什么少一半人重要？
    → 因为关系维度被压缩——没有爱慕、道侣、异性互动等最有戏剧张力的关系线
      → 为什么关系张力重要？
      → 因为 7game-lite 的核心价值是"观测弟子自发产生的故事"，关系越丰富故事越多
        → 根因价值：性别是叙事多样性的基础设施层
```

### 1.3 ROI 评估

| 维度 | 评估 |
|------|------|
| 开发成本 | M（7~8 文件改动，无新引擎） |
| 体验增量 | 4/5（沉浸感显著提升 + 解锁未来叙事可能） |
| 循环挂载 | 挂载现有弟子生成→AI 决策→MUD 输出主循环，不新增循环 |

### 1.4 最小组件

`LiteDiscipleState.gender` 字段 + 存档迁移。字段是一切下游的基础。

## §2 Invariant（不变量）

| # | 不变量 | 违反后果 |
|---|--------|---------|
| I1 | 性别不影响任何数值计算（灵气/突破/修炼速率/丹药） | 数值不公平 |
| I2 | 性别生成后不可变 | 叙事一致性崩溃，AI 上下文混乱 |
| I3 | 老存档升级后必须拥有合法性别值 | 存档读取崩溃或 undefined 穿透 |
| I4 | `'unknown'` 性别弟子使用中性代词"其"，不报错 | UI 文案出现 undefined/空白 |
| I5 | 性别仅作为 LLM prompt 上下文，不硬编码行为差异 | 未来扩展性被锁死 |

## §3 系统边界

### 3.1 IN — 本 Phase 范围

1. `LiteDiscipleState` 新增 `gender` 字段（`'male' | 'female' | 'unknown'`）
2. 弟子生成器：性别随机 + 性别化名字池
3. 代词系统：`getPronoun(gender)` + `{pronoun}` 填充
4. AI prompt：身份描述注入性别
5. MUD 面板：性别标识显示
6. 存档迁移 v6→v7
7. 需求债务清单登记

### 3.2 OUT — 明确排除

| 排除项 | 原因 | 登记为 |
|--------|------|--------|
| 外貌/容貌系统 | 独立大系统 | FB-021 |
| 爱慕/道侣关系机制 | 需独立 Phase 设计 | FB-022 |
| 繁衍/后代系统 | 依赖道侣系统 | FB-023 |
| 性别关联特性（好色、水性杨花等） | 需策划深度设计 | FB-024 |
| 性别锁定灵根/功法 | 违反 I1 倾向 | FB-025 |
| 碰面模板性别化文案 | 当前模板用 {A}{B} 足够 | FB-026 |

## §4 规则与数值边界

### 4.1 业务实体

| 实体 | 类型 | 说明 |
|------|------|------|
| `Gender` | union type | `'male' \| 'female' \| 'unknown'` |
| `FEMALE_RATIO` | 常量 | 女性生成概率，默认 `0.5` |
| `MALE_GIVEN_NAMES` | string[] | 男性名字池（≥15） |
| `FEMALE_GIVEN_NAMES` | string[] | 女性名字池（≥15） |
| `NAME_GENDER_MAP` | Record<string, Gender> | 现有名字→性别推测（迁移用） |

### 4.2 规则映射表

| # | 规则 | 输入 | 输出 |
|---|------|------|------|
| R-01 | 新弟子性别生成 | `Math.random()` vs `FEMALE_RATIO` | `< FEMALE_RATIO` → `'female'`，否则 `'male'` |
| R-02 | 性别→名字池 | gender | male→MALE_GIVEN_NAMES, female→FEMALE_GIVEN_NAMES |
| R-03 | 性别→代词 | gender | male→"他", female→"她", unknown→"其"（见 §4.3） |
| R-04 | 老存档迁移 | disciple.name 的 givenName 部分 | 查 NAME_GENDER_MAP → 匹配则用，无匹配→随机 |
| R-05 | AI prompt 注入 | gender | 身份描述追加"，性别男/女" |
| R-06 | 性别不可变 | 生成后 | gender 字段约定级不可变（匹配 `initialMoral` 模式：无 setter，无运行时守卫；迁移函数为唯一写入点）。TDD 阶段决定是否需要运行时断言 |
| R-07 | 性别不影响数值 | 任何数值公式 | 0 个公式引用 gender |

### 4.3 代词映射表（R-03 完整展开）

| gender | 主格代词 | 物主代词 | 反身代词 |
|--------|:-------:|:-------:|:-------:|
| `'male'` | 他 | 他的 | 自己 |
| `'female'` | 她 | 她的 | 自己 |
| `'unknown'` | 其 | 其 | 自身 |

## §7 Pre-Mortem

> 假设 Phase GS 上线后失败了，最可能的原因是什么？

| # | 失败场景 | 概率 | 预防措施 |
|---|---------|:----:|---------|
| PM-1 | 老存档迁移遗漏某弟子，gender=undefined 导致代词崩溃 | 中 | 迁移函数末尾断言：所有弟子 gender !== undefined |
| PM-2 | 名字池太小，8 弟子中出现重名 | 低 | 15×15=225 >> 8，且已有去重逻辑 |
| PM-3 | AI 过度解读性别，产出不当内容 | 中 | prompt 仅注入"性别男/女"，不加额外引导词 |
| PM-4 | 用户期望看到爱慕/道侣但本 Phase 没做 | 低 | PRD §3.2 明确 OUT + 需求债务登记 |
