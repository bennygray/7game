## 1. Gemini 测试方案的根本性错误诊断

Gemini 在 poc-2a/2b 中得出"0.8B 无法做行为决策，逻辑准确率 17%~25%"的结论。  
**该结论不成立。** 根本原因是 API 调用方式的技术错误，而非模型能力不足。

### 1.1 错误的做法（poc-2a/2b）

| 维度 | Gemini 的做法 | 问题 |
|------|--------------|------|
| API 端点 | `/completion`（裸文本补全） | 无法利用 Chat Template 结构化对话 |
| 格式约束 | 手拼 `<\|im_start\|>` 标签 + 正则提取 `<Decision:X>` | 模型必须自己"记得"输出格式，0.8B 做不到 |
| 参数 | `temperature: 0.1`, `repetition_penalty: 1.15` | 过低温度+惩罚项导致输出死板或进入死循环 |
| Few-Shot | 同语境修仙示例 | 造成"标签绑架"——模型无脑复制示例中的选项 |

### 1.2 正确的做法（poc-1c 已验证，poc-2c/2d/2e 延续）

| 维度 | 正确做法 | 原理 |
|------|---------|------|
| API 端点 | `/v1/chat/completions`（OpenAI 兼容 Chat API） | llama-server 自动应用 Qwen chat template |
| 格式约束 | `response_format: { type: 'json_schema', json_schema: { schema } }` | **底层编译 GBNF grammar，从 token 采样阶段就锁死输出空间** |
| 候选锁定 | `action: { type: 'string', enum: ['ATTACK', 'FLEE', ...] }` | 模型物理上只能输出 enum 内的值，100% 格式正确 |
| 温度 | `0.6` | 给模型足够的推理空间 |

> **核心认知：** `json_schema` 不是"提示词技巧"，它是 llama-server 的**硬件级约束机制**。服务端根据 schema 编译出 GBNF 语法树，在每一步 token 生成时 mask 掉所有不合法 token。模型不需要"理解" JSON 或"记住"格式——**它在物理层面被迫只能走合法路径**。

## 2. 测试演进全记录

### 2.1 poc-2c：技术路线修正验证（6 场景）

**目标**：验证使用正确的 API 端点 + JSON Schema 后，0.8B 能否做行为决策。

| 指标 | 结果 |
|------|------|
| 行为决策合理度 | **4/6 (67%)** |
| JSON 格式正确率 | **6/6 (100%)** |
| 平均延迟 | 376ms |

**结论**：0.8B 可以做行为决策。67% 的准确率虽不完美，但对照组表现优异（刚烈选 ATTACK、懦弱选 FLEE），证明模型具备性格→行为的映射能力。

### 2.2 poc-2d：大规模矩阵验证（20 场景 × 10 组对照）

**目标**：扩大样本量，覆盖 8 种事件类型，确认稳定性。

| 指标 | 结果 |
|------|------|
| 行为决策合理度 | **14/20 (70%)** ✅ |
| JSON 格式正确率 | **20/20 (100%)** |
| 对照组完全正确 | 5/10 (50%) |
| 平均延迟 | 425ms，P95 = 955ms |

**事件类型表现**：

| 事件类型 | 准确率 | 评定 |
|---------|--------|------|
| theft（偷窃） | 100% | 🏆 |
| cultivation（修炼竞争） | 100% | 🏆 |
| romance（情感） | 100% | 🏆 |
| alchemy（炼丹） | 100% | 🏆 |
| combat（战斗/落井下石） | 50% | ⚠️ |
| social（权力关系） | 50% | ⚠️ |
| sacrifice（恩怨） | 50% | ⚠️ |
| betrayal（背叛） | 50% | ⚠️ |

**发现的问题**：模型在同一次调用中同时做"行为决策"和"独白生成"时，两个任务互相干扰。典型症状：独白内容与选择的行为自相矛盾。

### 2.3 poc-2e：双阶段 Pipeline 验证（决策与独白分离）★ 定案

**目标**：验证将 Call 拆分为"纯决策 → 独白渲染"两步后的效果。

| 指标 | poc-2d (合体) | poc-2e (拆分) | 变化 |
|------|-------------|-------------|------|
| **行为决策合理度** | 14/20 (70%) | **15/20 (75%)** | **+5% ↑** |
| JSON 格式正确率 | 100% | 100% | 持平 |
| 对照组完全正确 | 5/10 (50%) | 5/10 (50%) | 持平 |
| 决策延迟 | — | **avg=304ms** | — |
| 独白延迟 | — | **avg=334ms** | — |
| 总延迟 avg | 425ms | 638ms | +213ms |
| 总延迟 P95 | 955ms | 1084ms | +129ms |

**拆分后修正成功的 case（poc-2d 错 → poc-2e 对）**：

| ID | 角色 | poc-2d 错选 | poc-2e 正确 | 原因 |
|----|------|-----------|-----------|------|
| T02 | 苏瑶（懦弱） | ATTACK | **ENDURE** | 不再被独白中的"勇气"词汇污染 |
| T16 | 老神仙（佛系） | SCOLD | **FORGIVE** | 纯决策阶段不受"训斥独白"的反向干扰 |
| T20 | 陈明月（温和） | EXECUTE | **COMFORT** | 决策不再被"正义独白"带偏 |

**独白质量对比示例**：

| 角色 | poc-2d 独白（合体生成） | poc-2e 独白（分离生成） |
|------|----------------------|----------------------|
| 乌龟老祖 | "身为外门弟子...若我拔剑...不仅会暴露修为...此刻苟活于肉身..." (97字) | **「苏瑶归途，我亦如草芥。贪生怕死，苟且偷生。」** (19字) |
| 狂浪生 | "作为刚烈如火的狂浪生...若今日她死...即便修为仅炼气三层..." (89字) | **「金丹长老欲夺苏瑶，我唯愿为苏瑶代尝毒草，以护其性命，方不负此生情种之誓。」** (35字) |
| 老神仙 | "老神仙虽慈悲，但面对弟子顶撞掌门...与其强行罚她闭关..." (65字) | **「弟子顶撞我，我虽慈爱，却终见慈悲。若计较，便是凡人；若随缘，便是大道。」** (32字) |

**规律**：拆分后的独白更短、更有仙侠韵味、更贴合性格。合体模式下模型倾向于生成冗长的"解释性"文本来合理化自己的选择。

## 3. 仍存在的顽固失败模式

以下 case 在 poc-2d 和 poc-2e 中**均失败**，属于 0.8B 模型的硬性偏置：

| ID | 角色 | 期望 | 实际 | 原因分析 |
|----|------|------|------|---------|
| T03 | 李墨染（阴险毒辣） | ATTACK/LOOT | REPORT | 模型的"正能量偏置"——训练数据中"救人/报告"远多于"杀人/抢劫" |
| T15 | 赵铁柱（军人气质） | PUNISH/SCOLD | FORGIVE | 模型倾向"文明选项"，不愿生成惩罚性行为 |
| T14 | 张清风（恩怨分明） | THANK/QUESTION | REJECT/ATTACK | 模型对"死敌救命"这种复杂矛盾场景处理能力不足 |

**应对策略**（按实施成本排序）：
1. **Few-Shot 示例**：在 messages 中预置 2-3 个已答好的"反派选择恶行"示例，成本极低
2. **候选池精细化**：引擎层面根据 moral=evil 的角色，去掉 HELP/FORGIVE 等明显不符合人设的选项
3. **LoRA 微调**：制作 300~500 条训练数据覆盖反派行为模式，属于远期方案

## 4. 全版本对比总表

| 版本 | 技术路线 | 格式正确率 | 逻辑准确率 | 延迟 avg |
|------|---------|-----------|-----------|----------|
| poc-1b | Chat API + json_schema, 15 种情绪自由选 | 100% | **45%** | 656ms |
| **poc-1c** | Chat API + json_schema, **4 种情绪候选池** | 100% | **90%** | 639ms |
| poc-2a (Gemini) | /completion + 正则 | 17% | 17% | 106ms |
| poc-2b (Gemini) | /completion + FewShot | ~80% | 25% | ~100ms |
| poc-2c | Chat API + json_schema, 4 种行为候选 | 100% | **67%** | 376ms |
| poc-2d | Chat API + json_schema, 同上×20 场景 | 100% | **70%** | 425ms |
| **poc-2e** ★ | **双阶段分离: 决策(json_schema) + 独白(json_schema)** | 100% | **75%** | 638ms |

**规律提炼**：
- 候选池从 15→4：准确率从 45%→90%（情绪任务）
- 单次合体→双阶段分离：准确率从 70%→75%（行为任务），独白质量大幅提升
- 行为决策（75%）比情绪分类（90%）天然更难，因为涉及因果推理而非纯语义匹配

## 5. 最终架构定案：双阶段 Soul Pipeline

基于全部测试数据，确立以下生产架构：

```
╔══════════════════════════════════════════════╗
║           引擎 Tick 层（硬代码）                ║
║  ├─ 根据 GameState 生成合法行为枚举              ║
║  │   例: [ATTACK, FLEE, MEDITATE, ARGUE]       ║
║  ├─ 根据 事件×关系×道德 预筛情绪候选池            ║
║  │   例: [anger, sadness, contempt, joy]       ║
║  └─ 将 enum 注入 JSON Schema                   ║
╠══════════════════════════════════════════════╣
║           AI Call 1: 纯决策（~300ms）           ║
║  ├─ 输入: 结构化 Prompt (事件+角色+关系)         ║
║  ├─ Schema: { action: enum, confidence: 1-3 }  ║
║  ├─ 特点: 极简输出，模型 100% 算力做逻辑推理     ║
║  └─ 约束: json_schema enum 底层锁死              ║
╠══════════════════════════════════════════════╣
║           AI Call 2: 独白渲染（~330ms）          ║
║  ├─ 输入: 事件 + 角色 + 已确定的行动             ║
║  ├─ Schema: { innerThought: string }            ║
║  ├─ 特点: 纯文学创作，模型全力发挥想象力          ║
║  └─ 已知行为 → 独白围绕行为展开，不会自相矛盾     ║
╠══════════════════════════════════════════════╣
║           引擎执行层（硬代码）                    ║
║  ├─ 解析 action → 调用对应的 GameState 变更      ║
║  ├─ 情绪系统（已由 poc-1c 验证） → 更新情绪       ║
║  └─ innerThought → 渲染到 MUD 日志              ║
╚══════════════════════════════════════════════╝
```

### 5.1 该架构的核心优势

1. **职责单一原则**：每次 AI 调用只做一件事，0.8B 的有限算力不被分散
2. **100% 格式安全**：json_schema enum 在物理层面保证输出合法，不依赖 Prompt 祈祷
3. **独白质量高**：已知行为后生成独白，风格简洁、贴合性格、不会自相矛盾
4. **延迟可控**：双阶段总计 ~638ms，远低于 1s 的游戏实时性预算
5. **引擎可兜底**：所有游戏逻辑的正确性由引擎的 enum 边界保证，AI 选错了也不会导致非法状态

### 5.2 与情绪系统的统一

| 维度 | 情绪系统 (poc-1c 验证) | 行为系统 (poc-2e 验证) |
|------|---------------------|---------------------|
| 候选池来源 | 引擎根据事件×关系预筛 4 个情绪 | 引擎根据 GameState 生成 4 个合法行为 |
| Schema 约束 | `emotion: { enum: [...] }` | `action: { enum: [...] }` |
| 准确率 | **90%** | **75%** |
| 延迟 | ~640ms | ~300ms (仅决策) |

两个系统可以**并行调用**，同一个 Tick 内同时获取情绪和行为，总延迟不叠加。
