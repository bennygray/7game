## 一、设计原则

### 1.1 核心理念：建议选项 + 开放补充

**不采用纯开放式提问**。原因：
- 纯开放式（"你想做什么？"）认知负担高，USER 需要从零组织回答
- 纯封闭式（"A 还是 B？"）限制思路，可能遗漏 USER 的独特想法
- **混合模式**：AI 基于项目上下文给出 2-3 个建议选项 + "或者你有其他想法？"

**信息流对比**：

| 模式 | 示例 | 问题 |
|------|------|------|
| 纯开放 | "这个系统的边界是什么？" | USER 不知从哪说起 |
| 纯封闭 | "边界是 A 还是 B？" | 遗漏 C/D/E |
| **建议+开放** | "边界方面，基于现有架构我建议：(a) X (b) Y (c) 其他——你的想法？" | 降低门槛 + 保留自由度 |

### 1.2 建议选项的来源

AI 不能凭空编造选项。选项必须基于：
1. **项目文档**：MASTER-PRD、已有 Phase 的 PRD/TDD
2. **代码现状**：grep 搜索相关模块的实际结构
3. **前序 Phase 模式**：类似系统过去怎么做的

如果 AI 对某维度缺乏信息无法给出有意义的选项 → **退化为开放式提问**（允许但需标注原因）。

## 二、触发 / 跳过条件

### 2.1 触发（二选一命中即触发）

| 条件 | 判定方式 |
|------|---------|
| USER 输入少于 3 句话且涉及**新系统** | AI 判断：输入中包含"新功能/新系统/新机制" + 输入 < 3 句 |
| USER 表达不确定 | 关键词："不确定"、"大概想做"、"有个想法"、"还没想清楚" |

### 2.2 跳过

| 条件 | 判定方式 |
|------|---------|
| USER 已给出详细需求 | 输入 ≥ 10 句话，或已附 Brief/PRD 草稿 |
| 纯 bugfix / 小改动 | AI 判断：修复类、调参类、UI 微调类 |
| USER 明确拒绝 | "跳过采访"、"直接分析"、"不用问了" |

### 2.3 中途退出

USER 可随时说"够了，开始分析"→ AI 立即整理已有信息，跳入 Step 1。

## 三、执行规则（SKILL.md 拟写入内容）

以下为计划写入 SPM SKILL.md 的完整文本（Step 0 和 Step 1 之间插入）：

```markdown
## Step 0.5：需求采访（Spec Interview）— 条件触发

> **定位**：在 Step 1 之前，通过定向提问提高需求输入质量。
> **模式**：建议选项 + 开放补充（非纯开放式提问）。

### 触发条件（二选一）

- USER 输入少于 3 句话且涉及新系统/新机制
- USER 明确表达不确定（"大概想做"、"有个想法"等）

### 跳过条件

- USER 已给出详细需求描述（≥10 句话或已有 Brief）
- 纯 bugfix / 小改动 / 参数调整
- USER 明确说"跳过采访" / "直接分析"

### 执行纪律

1. **一次只问 1 个问题**——禁止连珠炮式提问
2. **先问后整理**——不得先输出分析再让 USER 确认（防止橡皮图章）
3. **建议选项优先**——每个问题给出 2-3 个基于项目上下文的建议选项 + 开放补充入口
4. **选项必须有据**——选项来源于项目文档/代码现状/前序 Phase 模式，禁止凭空编造
5. 如某维度 AI 缺乏信息无法给出有意义选项 → 退化为开放式提问，标注"[信息不足，开放提问]"
6. 允许 USER 回答"不确定"——标记 [待决]，不追问
7. **3-5 轮后**整理结构化摘要交 USER 确认
8. USER 随时可说"够了"→ 立即整理已有信息，进入 Step 1

### 问题框架（按顺序，按需跳过）

| # | 维度 | 建议选项式提问模板 |
|---|------|-------------------|
| Q1 | 核心体验 | "做完这个，玩家体验上的主要变化是？我看到几种可能：(a) [基于 PRD 的选项 A] (b) [选项 B] (c) 你有其他想法？" |
| Q2 | 系统边界 | "这个系统一定**不做**什么？参考 [相关现有系统]，我建议排除：(a) [排除项 A] (b) [排除项 B] (c) 其他边界？" |
| Q3 | 触发方式 | "什么情况下触发？(a) 系统自动（类似 [现有自动触发系统]）(b) 玩家指令触发 (c) 混合 (d) 其他？" |
| Q4 | 系统关系 | "它和 [最相关现有系统] 的关系是？(a) 扩展——在其基础上增加 (b) 替代——取代现有 (c) 并行——独立运作但数据互通 (d) 其他？" |
| Q5 | 成功标准 | "怎么判断它做成了？(a) [可观察行为 A] (b) [可观察行为 B] (c) 你的标准？" |

> **[方括号]** 内容由 AI 根据项目上下文实时填充，不是字面输出。

### 产出

- **需求摘要**（≤ 10 行）——整合 USER 的回答
- **[待决] 列表**——USER 回答"不确定"的项
- **选项决策记录**——USER 选了哪个选项或自定义了什么
- → 整体作为 Step 1 的输入（替代 USER 的原始一句话）
```

## 四、示例对话（Phase I-alpha 假设场景）

假设 USER 只说了一句："我想做弟子之间的因果事件系统。"

### Step 0.5 实际对话流

**轮 1 — Q1 核心体验**：

> 做完因果事件系统，玩家体验上的主要变化是什么？基于现有的 soul-engine 和关系系统，我看到几种可能：
>
> (a) 弟子之间的互动不再只是数值变化（关系分+1），而是会产生**可阅读的事件文本**（类似世界事件但由弟子关系驱动）
> (b) 玩家能看到弟子关系的**因果链**——A 和 B 为什么从朋友变成了对手，有完整的事件记录
> (c) 其他想法？

USER："(a) 和 (b) 都要，而且事件应该影响弟子的后续 AI 决策。"

**轮 2 — Q2 边界**：

> 了解。这个系统一定**不做**什么？参考现有的 world-event 系统（全局事件漏斗），我建议排除：
>
> (a) 不替代 world-event——因果事件是弟子级别的，world-event 是世界级别的
> (b) 不做实时对话——事件以 MUD 日志形式呈现，不需要玩家参与对话
> (c) 其他排除项？

USER："同意 (a)(b)，另外不做多于 2 人的群体事件，至少第一版不做。"

**轮 3 — Q4 系统关系**（跳过 Q3，因为触发方式已隐含在 USER 的回答中）：

> 它和 soul-engine 的关系是？
>
> (a) **扩展**——在 soul-evaluator 的 Lv.1~3 决策基础上增加"因果事件"作为新的决策触发源
> (b) **并行**——独立的因果引擎，通过 RelationshipMemory 与 soul-engine 共享数据
> (c) 其他？

USER："(a)，它应该是 soul-engine 的一部分。"

**摘要整理**：

> **需求摘要**（Step 0.5 产出）：
> 1. 因果事件系统是 soul-engine 的扩展，弟子级别
> 2. 产出可阅读的事件文本 + 因果链记录
> 3. 事件影响弟子后续 AI 决策
> 4. 不替代 world-event（世界级）
> 5. 不做实时对话，MUD 日志形式
> 6. 第一版限定 2 人事件
> 7. [待决] 成功标准（未讨论）
>
> 确认后进入 Step 1（5-Why）。

## 五、与现有 SPM 流程的衔接

### 5.1 流程变化

```
当前：
  USER 想法 → Step 0（项目检测） → Step 1（5-Why + 价值锚定） → Step 1.5（PoC） → ...

改后：
  USER 想法 → Step 0（项目检测） → [Step 0.5 采访（条件触发）] → Step 1（5-Why，输入更充分） → ...
```

### 5.2 SKILL.md 插入位置

在现有 `Step 0` 段落结束后、`Step 1` 段落开始前插入。具体位置：

```
SKILL.md L70（Step 0 末尾："此步骤自动执行，不计入 Gate"）
→ 插入 Step 0.5 全文
SKILL.md L73（Step 1 开头："## Step 1：第一性原理解构 + 价值锚定"）
```

### 5.3 不影响的部分

- Step 1 的 5-Why / Invariant / 最小组件 — 不变
- Step 1.5 的 PoC 触发条件 — 不变
- Step 2 及之后 — 不变
- Gate 1 签章 — 不变

### 5.4 Step 0.5 产出如何流入 Step 1

| Step 0.5 产出项 | Step 1 消费方式 |
|----------------|----------------|
| 需求摘要 | 作为 5-Why 的输入（替代 USER 原始一句话） |
| [待决] 列表 | 5-Why 过程中尝试通过根因分析解决，解决不了保留到 Step 2 |
| 选项决策记录 | 作为 Invariant 声明的候选输入（边界类选项 → 不变量） |

## 六、风险与缓解

| 风险 | 缓解 |
|------|------|
| 建议选项引导 USER 思路，限制创新 | Q 模板都有"其他想法？" 出口；选项基于已有项目上下文，不是凭空限制 |
| AI 给出的选项质量差（不了解项目） | 选项必须有据（文档/代码/前序模式）；缺乏信息时退化为开放式 |
| 3-5 轮对 USER 来说太多 | USER 随时可说"够了"；跳过条件宽松，很多场景不触发 |
| 讨好偏差（NN/g 警告） | 不做苏格拉底式假设挑战；选项中包含"排除项"维度天然带有建设性质疑 |

---

## 七、不做的事情（显式排除）

以下 SSA 设计中的机制**不纳入** Step 0.5：

| 机制 | 排除理由 |
|------|---------|
| 苏格拉底假设挑战 | 讨好偏差——AI 挑战人类假设效果存疑（NN/g） |
| 复杂度分级 Lite/Standard/Full | 增加判断成本，二元触发/跳过已够用 |
| 假设映射 2×2 矩阵 | SPM Step 1 的 Pre-Mortem 已覆盖 |
| Appetite 投资预算 | 超出 SPM 职责范围（属于项目管理层） |
| Living Spec 回更 | 已有 handoff.md 机制，不重复建设 |

## 八、验证方法

实施后，通过以下方式验证 Step 0.5 的效果：

1. **首次验证**：在下一个新系统 Phase 的 SPM 环节中实际执行 Step 0.5，观察：
   - 采访产出的需求摘要 vs USER 原始输入的信息增量
   - Step 1 的 5-Why 是否因输入更充分而更聚焦
   - USER 的主观感受（有帮助 / 浪费时间 / 需要调整）
2. **长期指标**：PRD v2/v3 重写频率是否下降（Phase IJ 式的 v1→v2→v3 是否减少）
